# AI Coding 双工协作模式解决方案

> 针对 problem.md 中"一人主开发 + 一人主修复、各有独立 agent"场景的分支规范与协作模式设计。分场景的操作步骤见配套手册 `handbook.md`。

## 一、问题本质与总体思路

三个根因：

1. **从分支寿命太长**：差异随时间累积，最终合并变成靠 AI"读懂再融入"的重写。
2. **两个 agent 没有共享记忆**：彼此不知道对方改了什么；仓库是唯一双方都能看到的媒介，但没被当作通信信道来用。
3. **质量防线与验证环境错位**：检查在外网、实测在内网，外网检查覆盖不了内网特有的问题，失败案例也没有回流成外网可跑的测试。

对应的三个原则：

- **main 是唯一事实来源**：两条线都从 main 拉、向 main 合；main 不直接承载开发提交，只接收验证通过的分支合入。
- **小而频繁代替大而久拖**：功能拆小分支，修复走短命分支，双向同步以天为单位。
- **仓库即共享记忆**：spec、变更日志、约定全部入库，agent 开工先读仓库，口头同步只作补充。

## 二、分支模型

```
main（受保护，只接收验证通过的合并）
 ├── fix/<bug>     修复分支：单一职责，实测通过后合回
 ├── feat/<功能>   功能分支：≤3~5 天，每天 merge main
 └── 不再设长期"从分支"
```

1. **main 受保护**：bug 修复也走 `fix/*` 分支，不直接落在 main 上。
2. **取消长期从分支**：新功能按 spec 拆成可独立交付的最小单元，做不完就拆成多个 spec 分批合入。
3. **每日同步**：main 每有新合入，开发 agent 当天 `git merge main` 进 feat 分支。全团队统一用 merge 不用 rebase（不改写历史、无需 force-push、hash 稳定便于内外网对账）。冲突在差异小时解决，成本最低。
4. **fix 分支单一职责**：一个分支只修一个 bug；因实测往返可存活数天，但范围扩大就拆分。
5. **合并方向单向**：只允许 `fix/*`、`feat/*` → main，禁止分支间互合，横向同步一律经过 main。

### 分支删除规范

- **立即删**：已合入 main 且 diff 校验为空的分支（合并时当场删，本地+远程）；被替代的重复尝试。
- **归档后删**：决定放弃的未合并分支，先在 spec/issue 记录 commit hash、废弃原因、可复用片段再删——避免孤儿 commit 被 git 回收后无处找回。
- **禁止删**：迭代验证环中的分支（内网所测代码的唯一出处）；未合并未归档的分支；被未关闭 spec/issue 引用的分支；main。
- **纪律**：谁的分支谁删、不留尾巴；每周检查存活超 7 天的分支，逐个决定继续迭代、合并或归档删除；分支不复用不复活（重开用 `fix/xxx-2`）；内网临时分支实测出结论即删，内网只长期保留 main 和 tag。

## 三、两个 Agent 的协作协议

agent 之间无法直接通信，仓库是唯一的共享记忆，约定如下。

### 1. 开工引导

每次会话开始固定执行：读 `AGENTS.md`（本协议的仓库版）→ 读相关 spec → `git log --oneline main -20` 和 `git diff <基点>..main --stat` 看落后期间 main 的变化，需要细节再 `git show <hash>`。几分钟成本，换来"对方改了什么"完全可见。

### 2. 合入即公告

- **commit message 即公告**（不单独维护 CHANGELOG）：统一 `type(scope): 摘要` 格式，type ∈ `feat` / `fix` / `refactor` / `config`；fix 的正文写根因，涉及对外行为、接口、数据结构变化的必须在正文写明。squash 合入保证 main 上一个 commit 对应一件完整的事，`git log` 就是变更记录，agent 按 type 前缀筛选、按需 `git show`。
- **feat spec 中**：必须声明会改动/依赖哪些既有模块，等于提前向修复方公告冲突域。
- 接口（API、库表结构、共享类型）变化集中记录在固定位置（本项目为 `docs/design-docs/`），双方 agent 必读。

### 3. 模块归属与冲突热点

- 新功能尽量**新增文件**，bug 修复**只改既有文件**，两条线在文件层面天然错开。
- 冲突热点文件（路由注册、全局配置、公共类型等）按模块拆分；无法拆分的，改动方合并后立即群内公告，另一方当天 merge main。
- 每天开工前 5 分钟互通"今天动哪些模块"，重叠则人工错开。

### 4. 问题归属判定

- **先看代码位置**：问题在未合入 main 的 feat 分支上 → 不是 bug，是开发未完成，开发方在原分支继续改，不进修复队列；问题在 main 上可复现 → 是 bug，走 fix 分支。
- **再看上下文深度**：main 上的 bug，局部缺陷归修复方；设计性缺陷可派回原开发者，但仍走 fix 流程，只是执行人换成最懂该功能的人。
- **铁律**：修复线只接受 main 上的可复现异常；同一时刻一个分支只有一方在改（帮忙可以写测试、审 spec，不进对方分支改代码）。

### 5. 重构类需求的特别约定

重构改的全是既有文件，无法与修复线"错开文件"，改为"错开时间 + 切片"：

1. **择时**：公告"X 模块重构窗口"，避开修复高峰；窗口内紧急 bug 修复优先，重构分支 merge main 吸收后继续。
2. **切小**：每片 1~2 天、独立验证独立合并；禁止憋大重构分支——那是人为再造一个长期从分支。
3. **行为不变且可证**：spec 声明"对外行为不变"，验收 = 既有测试原样全绿 + 内网回归实测；要改行为就拆成两个分支分别走流程，严禁混在一个 commit，否则实测出问题无法归因。
4. **修复先行**：重构中的模块出 bug，修复从 main 先合，重构分支当天 merge main 吸收；绝不在重构分支里顺手修 bug。

## 四、标准操作流程

两条线共用同一个**迭代验证环**：同步分支 → 内网 latest 实测 → 失败则把复现步骤、日志、截图完整回传给 agent → 原分支上继续改 → 再同步。全程不碰 main，中间 commit 留在一次性分支上，main 不受污染。

### 修复线

1. 从最新 main 拉 `fix/<bug>`；spec 写现象、根因、方案、影响面；
2. 修复 + 外网检查通过 → 进入迭代验证环，直到实测通过；
3. squash 合入 main（`git merge --squash`，把分支上所有迭代 commit 压成一个干净 commit，message 按第三节规范写清根因与影响面），校验 `git diff main <fix分支>` 为空，删分支（squash 合并后 git 不认合并关系，用 `git branch -D`）；
4. 打 tag 成为 stable 候选；失败案例回流为外网回归测试（第六节）。

### 开发线

1. 从最新 main 拉 `feat/<功能>`；spec 先行，声明触及的既有模块；
2. 每天 merge main 进 feat；完成功能单元、外网检查通过；
3. 合入前最后再 merge 一次 main 并在 feat 侧解完冲突 → 进入迭代验证环，直到实测通过；
4. 合入 main（同样校验 diff 为空），更新 spec 状态，删分支。

**合并动作两侧各自完成**（不用"同步外网 main"代替）：外网 squash 合入外网 main 后发 close 包；内网把内网分支（代码 + `[config]` commit）squash 合入内网 main，再对账。这样进入内网 main 的就是实测过的对象本身，`[config]` 配置变更也不会丢。

三个要点：

- **合入 main 的门槛 = 外网检查全绿 + 内网实测通过**（typo、文案、样式等低风险改动仍可外网检查通过后直接合入）。
- **合并责任在 feat 一侧**：懂功能的人消化 main 的变化，远比懂 main 的人反向理解一整个月的新功能便宜——原来的"AI 读从分支再融入"环节就此消失。
- **反馈质量决定迭代轮次**：回传的日志和复现步骤越完整，AI 收敛越快；"实测不通过"这半句话对 agent 毫无价值。

**操作方式：一律 git 命令，agent 执行、人对写操作确认，不用仓库网页界面**。创建分支、合并、打 tag、删分支都在本地命令行完成——diff 校验、message 规范、close 包生成嵌在命令链里，人确认一次 agent 做完一串；界面点击没有这些闸口，容易跳步。人对每个写操作（合 main、删分支、打 tag）在 agent 汇报"实测通过 + diff 为空"后确认一次，这就是两人团队的评审环节。若合规要求合并留审批痕，改用平台 CLI/API（如 `glab` / `gh`）建 MR，仍是命令驱动，二选一不混用。main 在平台上配置：禁 force push、仅两名维护者可推、禁删除。内网侧全部走 `import.sh`，无界面操作。所有脚本的网络重试与原子化设计见第五节第 6 条。

## 五、内外网同步规范

安全规定不变，能改的是同步的**粒度、可靠性和自动化程度**。

1. **以 commit 为单位同步**：首选 `git format-patch` / `git bundle`（保留历史，不漏文件）；若规定只允许拷普通文件，则同步变更文件全集 + 清单（commit hash、文件列表、逐文件 SHA256），内网校验一致后才提交。
2. **批量同步**：按迭代轮次/合入事件触发，一天 1~2 次，每次是完整自洽的 commit 集合。
3. **同步包自包含，内网一键导入**。包分两类：sync 包携带分支最新代码（迭代验证环中用），close 包通知"分支已在外网合入 main"（收尾用）。内网侧的人工动作固定为三步：**拷 zip 到内网仓库一级目录 → 右键"解压到当前位置" → Git Bash 执行 `./import.sh`**（零参数、零配置、自动定位仓库、成功后自动清理包文件），其余全部自动化，操作者不需要知道包是哪种类型：

   ```
   sync-<分支名>-<序号>/
   ├── manifest.sh      # 可直接 source 的 KEY=VALUE 清单（类型、分支、base、上一包 tree 校验值、tip、tree 等）
   ├── message.txt      # commit message（close 包用）
   ├── configImpact.txt # 可选：部署影响待办，无则不生成
   ├── files.txt        # TSV 清单：action、git blob hash、path，按 path 排序
   ├── payload/         # 自上一包以来的增量 patch（首包全量），或变更文件全集
   └── import.sh        # 唯一入口：零参数，自读 manifest 分派处理
   ```

   内网执行环境是内网人员的 Windows 个人电脑 + Git Bash（只装了 git，无 jq 等工具），脚本只依赖 git 与基础 bash 命令；内容校验一律用 git 对象 hash（blob/tree），天然免疫 Windows 换行符（CRLF）差异。

   `import.sh` 零参数执行，自动完成全部动作（网络操作走内置重试通道、与 `tools/safe-git.sh` 同源，包自包含不依赖仓库内脚本；每步写日志，满足审计）：

   - **零配置自动定位仓库**：包解压到仓库一级目录，脚本向上找 `.git` 定位，执行人无需编辑脚本任何一行；成功后自动清理包文件，失败保留以便重跑；
   - **与执行时所在分支无关**：不要求人员处于任何特定分支。脚本开始先记录现场（当前分支、状态），随后用 `git worktree` 创建临时工作区，所有检出、应用、合并、校验都在临时工作区内对目标引用操作——不切换人员当前分支、不碰其未提交改动；结束（无论成败）移除临时工作区，人员现场原样恢复；
   - 读 manifest 自动分派：
     - **close 包** → 内网同名分支 squash 合入内网 main（message 随包给出，`[config]` commit 随之带入）→ 对账（与外网 main 排除 `deploy-intranet/` 后 diff 必须为空）→ 删除内网临时分支；
     - **sync 包** → 校验序号连续、目标分支当前 tree 与上一包 treeHash 一致（防漏包乱序；commit hash 因 committer 被 git am 改写而跨机不可比，一致性校验一律用 tree）→ 分支不存在自动从 base commit 建同名分支 → 应用 payload（patch 用 `git am`；文件模式按清单覆盖，删除只按 manifest 差集，永不触及保护路径）→ 逐文件 blob + 最终 tree 校验；
   - **重复执行安全（幂等按包类型判定）**：sync 包以"分支 tree 一致 + 远端 tip 一致"判定，远端落后自动补推送；close 包以"main tree 一致 + 分支已删"判定，main 已是目标内容时跳过合并直接收尾（半完成自愈）；判"已是最新"后同样自动清理包文件；
   - 成功后自动触发 latest 部署流水线（若规定部署必须人工确认，则输出唯一一条待执行命令）；任何一步失败自动还原（目标分支引用恢复执行前的值、临时工作区移除），并给出明确的下一步提示（如"main 同步落后，请先执行 sync-main-NNN 包"）。

   外网侧对应一个导出脚本，一条命令出包、序号自增；分支合入 main 后自动补发 close 包。**出包只能由导出脚本完成**：禁止手工组包、改包、脚本失败后绕过，manifest 字段与前置校验的强制规范见 `agent-pack/skills/sync-package`。若安全规定允许，内网可再进一步：设固定收件目录 + 定时任务，到包即自动执行，人工只剩"拷包"一步。

4. **同步源纪律**：latest 可部署任何分支的同步包；stable 只认 main 上打 tag 的 commit；合入 main 必须校验与已实测分支 diff 为空，保证"内网测过的"和"main 上的"内容等价。

5. **内网配置目录的边界**：内网仓库一级目录的部署配置文件夹 `deploy-intranet/` 属于"内网领土"，规则如下：

   - `deploy-intranet/` 外网与同步包永不包含该路径；`import.sh` 对触及它的操作直接拒绝；删除逻辑只按 manifest 差集，从机制上杜绝误删。
   - 内网分支 = 外网 commit + 追加在末尾的 `[config]` 内网 commit（不改写历史）。`git am`、bundle+merge 都动不了 `deploy-intranet/`；**唯一禁止**：用外网分支整体替换或 hard reset 内网分支。
   - 配置随功能调整：spec 固定"部署影响"一节 → manifest 的 `configImpact` 字段 → import 后打印待办，内网据此追加 `[config]` commit；latest 部署前检查未消化待办，有则禁止部署，挡住"代码到了、配置没跟上"的假失败。
   - 演进方向：配置拆"模板（外网随代码同步）+ 数值（留在 `deploy-intranet/`）"两层，内网只补数值。
   - `[config]` commit 随分支合入内网 main；tag 同时固化代码与配置，回滚 stable 时配置一起回滚。

6. **网络容错与原子化**（外网 GitHub、内网 Gitee 访问均不稳定，所有脚本统一遵守）：

   - **本地先成事实，网络只负责公布**：squash、合并、打 tag 先在本地完成并验证，网络操作只负责把已自洽的状态推上去。推送天然幂等（同 SHA 重复推等于无操作），重试永远安全，无需"回滚"。
   - **统一重试**：所有网络调用（push / fetch / ls-remote / API）走同一包装函数：每次调用设超时上限，失败后指数退避重试（如 5s / 15s / 45s / 2m / 5m，最多 5 次）；只对临时性错误重试（超时、连接重置、5xx），鉴权失败、404、权限拒绝立即报错，不盲目重试。
   - **操作顺序即回滚策略**：合 main 的顺序固定为 本地 squash → 本地 diff 校验 → push main 并确认远端已包含 → 推 tag（多引用推送用 `git push --atomic`）→ 全部确认后才删分支。任何一步网络失败就停在原地重试，分支和 tag 都在，什么都不丢；分支删除永远排在所有校验通过之后。
   - **断网挂起而非失败**：持续不可达时，脚本把待办操作写入本地队列（outbox）后退出，网络恢复后自动续做；agent 不得因一次 push 失败就重做整个合并流程。
   - **agent 只走包装脚本**：`AGENTS.md` 明文规定 agent 禁止裸用 `git push` / `git fetch`，一律调用 `tools/` 下的包装脚本，重试与校验逻辑全仓库只有一份。
   - **预检探活**：每个脚本开头 `git ls-remote` 探活，不通就直接提示稍后重试，不做任何本地改动。

## 六、质量防线：让外网检查逼近内网实测

1. **失败案例回流**：内网翻车先回答"为什么外网没查出来"，把场景转化为外网可跑的自动化测试（内网特有依赖用 mock/契约快照代替），测试集持续增厚，翻车率逐月下降。
2. **外网最低门槛**（合入 main 前全绿）：构建、类型检查、lint、单测、关键流程契约测试、spec 与代码一致性自查。
3. **latest / stable 纪律**：latest 部署最新同步 commit；stable 只前进到实测通过并打 tag 的 commit；回滚 = 重部署上一个 stable tag，秒级完成。
4. **环境差异清单**：列出内网有、外网没有的要素（数据库版本、浏览器、网络策略等），涉及这些要素的改动在 spec 标注"内网必测项"，latest 优先验证。

## 七、落地步骤

- 第 1 周：保护 main；建立分支命名规范；用 `agent-pack/templates/bootstrap.sh` 给老仓库一次性铺好 tools/、钩子、AGENTS.md 与 skills（spec 沿用现有 `docs/` 结构，映射已适配）；现存长期从分支拆分——能合的整理成小 feat 分支尽快合入，其余重做或放弃。
- 第 2 周：落地同步包脚本（外网导出 + 内网导入）；落地 latest→stable 的 tag 与 checklist 流程。
- 第 3 周起：执行失败案例回流，每次内网翻车至少沉淀一个外网可跑的测试；每两周回顾冲突热点文件，该拆的拆。

## 八、模式总览

```
开发 agent                            修复 agent
    │  拉 feat 分支                      │  拉 fix 分支
    │  每天 merge main                  │  修复 + 外网检查
    ├───────────► 迭代验证环 ◄───────────┤
    │    同步分支 → latest 实测           │
    │    → 回写日志 → 原分支再改          │
    ▼                                   ▼
     实测通过 → 两侧各自合入 main → 对账
              （排除配置目录后 diff 为空）
                      │
                打 tag → stable 前进
                      │
                失败案例回流为外网测试
```

核心收益：合并从"AI 重读重写"变成"日常小步同步、main 侧零冲突"；两个 agent 以仓库为共享记忆；多轮实测迭代收敛在一次性分支上，main 只接收验证通过的干净结果；内外网同步自动化、可校验；内网翻车的教训持续沉淀为外网防线。
