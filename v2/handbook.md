# 双工协作模式 v2 手册

> v2 相对 v1 的主题是**简化**：AI 退出同步过程，人重新成为分支管理的第一责任人。
> 本手册面向人（主开发 / 主修复）。AI 是你的助手，不是流程的看管者。

## 一、v2 与 v1 的区别

| 维度 | v1 | v2 |
| --- | --- | --- |
| 内外网同步 | AI 生成增量同步包 + import.sh（复杂、易错、耗 token） | **固定同步工具 + GitHub 官方全量 zip**（无 AI、确定性） |
| 分支管理 | 依赖 AGENTS.md / skill 约束 agent | **人是第一责任人**，规则写在本手册，自觉遵守；AI 可被安排代劳 |
| 远程网络 | 脚本重试 / outbox / 代理约定 | **人自行处置**（代理、重试、择时） |
| 内网提交记录 | 与外网逐 commit 对应（靠 patch 重放） | **不保证一致**：每次同步是工具生成的一个 commit，但内容（tree）一致 |
| 项目改造成本 | 需要 AGENTS.md / skills / tools 全套 | **几乎为零**：同步是纯脚本，分支靠人，skill 只是可选辅助 |

## 二、三条原则

1. **人是分支管理第一责任人**：建分支、同步 main、合并、删分支，规则在第四节，自己遵守（可以安排 AI 代劳，但责任在人）；
2. **同步无 AI**：内外网之间唯一的代码通道是"GitHub 网页下载的 zip + 同步工具"，纯脚本、全量覆盖、不看变更内容；
3. **网络自理**：pull/push/下载失败，自己重试、换代理或择时，不依赖脚本。

## 三、目录布局

```
<工作目录>/
├── <内网仓库>/      git 仓库（Gitee 克隆）
├── sync.py          同步工具（不受 git 管控）
├── sync.ini         参数文件：主分支名
├── sync.ignore      忽略清单：内网专属、不与外网同步的路径
└── *.zip            从 GitHub 下载的全量包（用完可删）
```

## 四、分支管理规则（人自觉遵守）

1. **main 不直接改**：新功能走 `feat/<名>`，修复走 `fix/<名>`，从最新 main 拉短分支；
2. **合并前先同步**：合并进 main 之前，先把 main 的最新变更 merge 进自己的分支并解完冲突；
3. **小步快走**：分支寿命以天计，功能拆小分批合；
4. **合并后删分支**（本地+远程）；
5. commit message 用 `type(scope): 摘要`（type ∈ feat/fix/refactor/config）；
6. **内网 main 不直接开发**：内网仓库只承载外网同步来的内容和忽略清单保护的内网专属文件；内网自有改动只允许落在 `sync.ignore` 保护的路径里（否则下次同步会被全量覆盖抹掉）；
7. **latest / stable**：latest 部署被同步的分支用于实测；stable 只前进到实测通过并打 tag 的 commit，回滚 = 重部署旧 tag；
8. 网络波动（pull/push/下载失败）：自己择时重试或挂代理，不找工具。

## 五、日常流程

### 新功能 / 修 bug（同一条路径）

1. 在外网仓库从最新 main 建分支（自己操作，或让 AI 建）；
2. 开发/修复（自己写或 AI 写），**在外网跑测试通过**；
3. GitHub 网页切换到该分支 → Code → Download ZIP，拷到内网同级目录；
4. 运行同步工具（见第六节），内网分支就绪；
5. 部署 latest 实测：
   - 有问题 → 回外网改 → 重新下载 zip → 再跑同步工具（幂等，增量变更只体现在 diff）→ 再测；
   - 通过 → 在外网把分支合并进 main（先 merge main 解冲突），删分支；
6. **main 同步**：下载 main 的 zip，对 main 跑一次同步工具，内网 main 对齐；
7. 到 stable 节点：实测没问题后在内网 main 上打 tag，部署 stable。

> AI 的位置：可以帮你写代码、跑测试、操作 git（在 1、2、5 步），但**同步这一步永远只是 zip + 脚本**，不让 AI 理解和打包变更。

## 六、同步工具使用

```bash
python3 sync.py <zip文件> --branch <分支名> --message "变更描述" [--base <基线分支>] [--push] [--repo <路径>]
```

- `--branch`：目标分支；不存在则自动新建（`--base` 指定基线，缺省用 `sync.ini` 的主分支）；
- `--message`：一句话变更描述，写入内网 commit（格式 `[sync] <分支>: <描述>`）；
- `--push`：完成后推送到 origin（可省，推送本就由人负责）；
- 行为：完整覆盖目标分支（zip 里没有的文件会被**删除**），`sync.ignore` 中的路径一律不动；**仓库自身的 `.gitignore` 也会自动遵守**（用 git 引擎判定，本地产物如 node_modules、data 等不会被删除或误提交）——所以 `sync.ignore` 只需声明"内网特有且被 git 跟踪"的内容（如 `deploy-intranet/`）；幂等——同包重跑报告"无内容变更"；
- 前置要求：内网工作区干净、已配置 git 提交者身份、本机有 `python3`（Windows Git Bash 若无，用 `py -3` 或安装 Python）。

`sync.ini`（参数文件）：

```ini
[sync]
main_branch = main        # 新建分支时未指定 --base 的默认基线
repo = agent-pack-test    # 目标仓库目录名：同级有多个 git 仓库时必填
```

仓库定位优先级：`--repo` 参数 > `sync.ini` 的 `repo` 项 > 自动识别（同级仅一个仓库时）。

`sync.ignore`（忽略清单，按需增删）：

```
deploy-intranet/    # 内网部署配置目录（前缀匹配）
.sync-state/
outbox/
data/
*.log
```

## 七、FAQ

- **zip 从哪来？** GitHub 仓库页面左上角选分支 → 绿色 Code 按钮 → Download ZIP。下载慢/失败自己挂代理重试。
- **会覆盖内网部署配置吗？** 不会，只要在 `sync.ignore` 里。新增内网专属文件时记得补清单。
- **为什么内网 commit hash 和外网不一样？** v2 的设计取舍：不复制提交记录，每次同步生成一个新 commit；保证的是**内容（tree）一致**。对账方式：两侧 `git rev-parse <分支>^{tree}` 应相同。
- **同步后发现多了一堆"未提交修改"？** 工具完成后会切回原分支；若你当时停在目标分支上，执行 `git reset --hard <分支>` 刷新工作区即可。
- **推送失败？** 网络问题，自己择时重试或挂代理 `http_proxy=... git push`（v2 原则 3）。
- **sync.py 坏了找谁？** 找机制维护方修；它是纯脚本，修完直接替换文件即可，不涉及仓库。

## 八、与 v1 的关系

- v1 的 skill（AGENTS.md、七个场景技能、superpowers 副本）仍可作为 AI 的工作规范**可选使用**，但模式运转不再依赖它们——这正是"双工协作成为一种人的意志"的含义；
- v1 的同步包机制（export/import）正式废弃，由本工具替代；
- 老项目（如 chargeX）接入 v2 时：仓库内基本不用改，只需在内网同级目录放三件套（sync.py/sync.ini/sync.ignore），并遵守第四节规则。
